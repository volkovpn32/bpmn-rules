1. Контекстно-зависимая логика
- Область действия: потоки управления, развилки и условные события, поведение которых зависит от контекста процесса.
- Требование: Следует минимизировать использование элементов, меняющих поведение процесса в зависимости от контекста, и явно пояснять на диаграмме смысл проверяемых выражений в подписях на потоках управления.

2. Внешние неконтролируемые участники
- Область действия: участники процесса, на действия которых владелец процесса или организация не может повлиять.
- Требование: Такие участники должны отображаться как внешние участники в виде свернутого пула.

3. Именование задач глаголом
- Область действия: узлы с типом `task`.
- Требование: Название задачи должно содержать один или несколько глаголов в неопределенной форме совершенного вида, то есть отвечать на вопрос «что сделать?».

4. Краткость названия задачи
- Область действия: узлы с типом `task`.
- Требование: Название задачи не должно содержать подробного описания или особенностей исполнения задачи.

5. Именование таймерных событий
- Область действия: узлы с типом `timer event`.
- Требование: Название должно содержать конкретный интервал времени, дату, время или контекст с таким указанием, а также признак цикличности либо одноразового запуска.

6. Выбор типа развилки по смыслу
- Область действия: узлы с типом `gateway`.
- Требование: `Exclusive Gateway` следует использовать, когда должен быть выбран только один путь, `Parallel Gateway` — когда должны быть запущены все пути, `Inclusive Gateway` — когда одновременно могут быть истинны несколько условий, а `Event-based Gateway` — когда выбор определяется наступившим событием.

7. Межпроцессные сообщения против сообщений физического мира
- Область действия: узлы с типом `message event`, `send task` и `receive task`.
- Требование: Эти элементы нельзя использовать для `e-mail`, `SMS` и сообщений в чатах, потому что в `PMA` они предназначены для межпроцессного обмена сообщениями между участниками.

8. Отображение ролей без дорожек
- Область действия: элементы диаграммы, где требуется показать роль исполнителя.
- Требование: Роли следует отображать не дорожками, а одним из рекомендуемых способов, например цветовой индикацией, текстом внутри активности, аннотацией или средствами редактора.

9. Отображение информационных систем
- Область действия: элементы диаграммы, где требуется показать используемую систему.
- Требование: Системы следует отображать одним из рекомендуемых способов, например через `Data Store`, текст внутри активности, аннотацию или возможности редактора.

10. Отображение документов
- Область действия: элементы диаграммы, где требуется показать документы процесса.
- Требование: Документы следует отображать одним из рекомендуемых способов, например через `Data Object`, аннотацию или возможности редактора.

11. Ограничения времени без изменения поведения
- Область действия: задачи и действия, для которых важно показать срок исполнения.
- Требование: Если нарушение срока не меняет поведение процесса, `BPMN` в рамках `PMA` не предоставляет нативного способа показывать плановый или фактический срок исполнения только ради визуализации срока.

12. Старт процесса по запросу внешнего клиента
- Область действия: старт процесса, инициируемый внешним клиентом.
- Требование: Такой старт следует моделировать через `Message Start Event` или `None Start Event`.

13. Внешний клиент как черный ящик
- Область действия: взаимодействие процесса с внешним клиентом.
- Требование: Внешнему клиенту нельзя назначать задачи внутри нашей процессной модели; следует показывать только само взаимодействие, например через сообщения со свернутым внешним пулом.

14. Запуск другого процесса без ожидания результата
- Область действия: фрагменты процесса, где один процесс инициирует другой и не ждет ответ.
- Требование: В этом случае текущий процесс должен просто инициировать другой процесс и продолжить выполнение без ожидания результата.

15. Запуск другого процесса с ожиданием результата
- Область действия: фрагменты процесса, где требуется дождаться результата другого процесса.
- Требование: Для такого сценария следует использовать `Call Activity`.

16. Ветка доработки и исправления
- Область действия: согласовательные процессы и развилки принятия решения.
- Требование: Нельзя ограничиваться только исходами «согласовано» и «отклонено»; при необходимости должна быть предусмотрена отдельная ветка «требуется доработка».

17. Прерывание процесса в любой момент
- Область действия: процессы, где альтернативное завершение может быть инициировано в любой момент выполнения.
- Требование: Для таких случаев следует использовать `Event Sub-Process` с прерывающими стартовыми событиями.

18. Переиспользование типовых процессов
- Область действия: повторно используемые типовые фрагменты процесса.
- Требование: Такие фрагменты следует выносить в отдельные процессы и переиспользовать через `Call Activity`.

19. Информационные `e-mail`, `SMS` и `push`
- Область действия: действия по отправке уведомлений в информационных целях.
- Требование: Для таких уведомлений следует использовать `Service Task`.

20. Моделирование данных
- Область действия: описание структуры данных, связанной с процессом.
- Требование: `BPMN` не следует использовать для моделирования данных; для этого рекомендуются `ER-диаграммы` или `диаграммы классов`.
